# DPO 트레이닝 데이터 고도화 가이드

## 개요

DPO(Direct Preference Optimization)는 별도의 보상 모델(Reward Model) 없이 데이터셋 내의 선호도(Chosen vs Rejected)를 직접 학습하기 때문에, 데이터의 질이 곧 모델의 성능을 결정합니다.

---

## 1. 목표

DPO의 목표는 단순히 "더 좋은 답변을 많이 넣는 것"이 아니라, 같은 입력에 대해 어떤 답변을 선호해야 하는지 모델이 안정적으로 배우게 하는 것입니다.

| 목표 | 설명 |
|------|------|
| 선호 기준 명확화 | 어떤 답변이 왜 더 나은지 일관되게 학습 |
| 도메인 품질 향상 | 법률/컴플라이언스/계약 검토처럼 특정 업무에서 더 정확한 판단 유도 |
| 나쁜 습관 제거 | 과잉 확신, 허위 인용, 장황함, 근거 없는 결론, 형식 위반 감소 |
| 경계조건 학습 | 애매한 케이스, 정보 부족, 근거 부족, "판단 불가" 상황에서 올바른 대응 |
| SFT 한계 보정 | SFT가 "그럴듯한 답변 생성"은 하지만 선호 기준을 못 잡는 부분 보완 |

핵심은 **정답 생성 데이터가 아니라 선호 판단 데이터**라는 점입니다.

---

## 2. 해야 하는 것 (Best Practices)

### 2.1 chosen의 품질이 rejected보다 훨씬 중요함

DPO 데이터는 보통 `prompt / chosen / rejected` 구조입니다.

```json
{
  "prompt": "...",
  "chosen": "...",
  "rejected": "..."
}
```

흔한 실수는 rejected를 일부러 너무 나쁘게 만드는 것입니다. 실제로 중요한 것은 **chosen이 정말로 목표 품질에 가까운 답변인지**입니다.

좋은 chosen의 조건:

| 기준 | 좋은 chosen |
|------|------------|
| 정확성 | 사실/법령/근거가 틀리지 않음 |
| 근거성 | 판단 이유와 근거가 명확함 |
| 지시 준수 | 사용자가 요구한 포맷/범위/톤을 지킴 |
| 불확실성 처리 | 모르면 모른다고 함 |
| 도메인 스타일 | 법률 검토, 계약 리뷰, 컴플라이언스 문체에 맞음 |
| 실무 유용성 | 바로 사용할 수 있는 구조와 수준 |

### 2.2 rejected는 "나쁜 답변"이 아니라 "실제로 헷갈릴 만한 덜 좋은 답변"이어야 함

**나쁜 예:**

| prompt | chosen | rejected |
|--------|--------|----------|
| 법률 검토 요청 | 정상 답변 | "몰라요ㅋㅋ" |

이런 데이터는 학습 효율이 낮습니다. 너무 쉬운 비교라서 실전 성능 개선이 적습니다.

**좋은 rejected 유형:**

| rejected 유형 | 설명 |
|---------------|------|
| 그럴듯하지만 근거가 약함 | 문장은 자연스럽지만 법적 근거 부족 |
| 결론은 맞지만 형식이 틀림 | 사용자가 요구한 JSON/마크다운/섹션 구조 위반 |
| 근거는 있으나 판단이 과함 | "가능성이 있음"을 "위법"으로 단정 |
| 일부 조항 누락 | 중요한 예외/조건/기간/주체 누락 |
| 도메인 용어 오용 | 해지/해제/취소/철회 등 법률 개념 혼동 |
| citation hallucination | 존재하지 않는 조문/판례/문서 인용 |
| 불필요하게 장황함 | 핵심 판단보다 설명 과다 |
| 안전한 거절 실패 | 근거 부족한데도 확정 답변 |

### 2.3 "선호 기준표(Rubric)"를 먼저 만들어야 함

DPO 데이터 라벨러가 각자 감으로 chosen/rejected를 고르면 데이터가 망가집니다.

| 우선순위 | 평가 기준 | 설명 |
|---------|----------|------|
| 1 | 사실 정확성 | 법령, 계약 조항, 수치, 날짜, 주체가 맞는가 |
| 2 | 근거 충실성 | 결론이 근거에서 따라오는가 |
| 3 | 지시 준수 | 사용자가 요구한 포맷/범위/톤을 지켰는가 |
| 4 | 불확실성 처리 | 정보 부족 시 추정하지 않는가 |
| 5 | 도메인 실무성 | 법무팀/컴플라이언스팀이 바로 쓸 수 있는가 |
| 6 | 간결성 | 불필요한 장황함이 없는가 |
| 7 | 문체 | 제품에서 원하는 톤과 일관되는가 |

### 2.4 실패 유형별로 데이터를 설계해야 함

| 카테고리 | 고도화 목적 |
|---------|------------|
| 판단 정확도 | 위법/적법/위험/불명확 판단 개선 |
| 인용 정확도 | 조항, 문서, 근거 citation hallucination 감소 |
| 관련 조항 선택 | 관련 없는 조항 끌고 오는 문제 감소 |
| 형식 준수 | JSON, 표, 섹션, mandatory structure 준수 |
| 거절/보류 | 정보 부족 시 무리한 답변 방지 |
| 다국어/용어 | 한국어 법률 용어, 영문 계약 용어 매핑 |
| 장문 컨텍스트 | 긴 계약서/규정에서 누락 방지 |
| 사용자 의도 | 캐주얼한 질문을 법률 개념으로 재해석 |

### 2.5 "차이가 명확한 pair"와 "미세한 pair"를 둘 다 넣어야 함

| 난이도 | 예시 | 목적 |
|--------|------|------|
| Easy | chosen은 정상, rejected는 명백히 틀림 | 기본 선호 방향 학습 |
| Medium | 둘 다 그럴듯하지만 rejected가 일부 누락 | 실무 품질 개선 |
| Hard | 둘 다 좋지만 chosen이 더 정확/간결/근거 충실 | 고급 선호 기준 학습 |
| Adversarial | rejected가 더 자신감 있고 자연스럽지만 사실은 틀림 | hallucination 억제 |

실제로는 **Medium/Hard pair가 가장 중요**합니다.

### 2.6 rejected를 현재 모델 출력에서 뽑는 것이 좋음

추천 파이프라인:

1. 실제 사용자/평가 prompt 수집
2. 현재 SFT 모델 또는 base 모델로 여러 답변 생성
3. 실패 유형별로 rejected 후보 분류
4. 사람이 chosen/rejected 결정
5. 왜 chosen인지 reason 태깅
6. DPO 학습
7. eval에서 다시 실패 케이스 수집
8. 다음 DPO 데이터로 재투입

### 2.7 데이터마다 "왜 chosen인지" 메타데이터를 남겨야 함

```json
{
  "prompt": "...",
  "chosen": "...",
  "rejected": "...",
  "preference_reason": "chosen은 조항 제5조와 제8조를 모두 근거로 들었고, rejected는 제8조의 예외 조건을 누락함",
  "error_type": "missing_exception",
  "domain": "contract_review",
  "difficulty": "hard"
}
```

이유:
- 라벨 품질을 검수할 수 있음
- 성능 저하 시 어떤 데이터가 문제인지 추적 가능
- 카테고리별 eval과 연결 가능

## 3. 주의해야 하는 것 (Pitfalls to Avoid)

### 3.1 noisy preference가 가장 위험함

DPO는 pairwise preference를 그대로 학습하기 때문에, chosen/rejected 라벨이 흔들리면 모델이 잘못된 선호를 배웁니다.

| 위험 데이터 | 문제 |
|------------|------|
| chosen도 틀렸는데 rejected보다 덜 틀린 경우 | 모델이 틀린 답변을 선호함 |
| 라벨러마다 기준이 다른 경우 | 선호 방향이 흔들림 |
| 문체만 보고 chosen 선택 | 정확성보다 말투를 학습 |
| 긴 답변을 무조건 chosen | 장황함 선호 |
| 자신감 있는 답변을 chosen | 과잉 확신 강화 |
| citation 있는 답변을 무조건 chosen | 허위 citation 강화 |

**DPO에서는 애매한 pair를 넣는 것보다 버리는 게 낫습니다.**

### 3.2 chosen/rejected의 길이 차이를 조심해야 함

| 체크 항목 | 권장 |
|----------|------|
| chosen 평균 길이 | rejected와 과도하게 차이나지 않게 |
| 긴 chosen | 실제로 근거/구조가 좋아서 chosen인지 확인 |
| 짧은 chosen | 핵심 누락 없이 간결한지 확인 |
| rejected | 일부러 대충 짧게 만들지 않기 |

### 3.3 형식만 좋은 답변을 chosen으로 두면 안 됨

우선순위: **정확성 > 근거성 > 지시 준수 > 실무 유용성 > 간결성 > 문체**

JSON extractor처럼 형식이 절대적으로 중요한 태스크는 예외.

### 3.4 rejected를 너무 악의적으로 만들지 말 것

나쁜 rejected:
> 계약 해지는 그냥 마음대로 하면 됩니다.

좋은 rejected:
> 계약 해지는 가능하나, 별도의 통지 기간 없이 즉시 해지할 수 있습니다.

두 번째는 "통지 기간", "귀책 사유", "해지권 발생 조건"을 누락했기 때문에 실제 모델 개선에 더 도움이 됩니다.

### 3.5 SFT 데이터와 DPO 데이터를 혼동하면 안 됨

| 구분 | SFT | DPO |
|------|-----|-----|
| 목적 | 정답 스타일 학습 | 선호 기준 학습 |
| 데이터 | prompt + ideal answer | prompt + chosen + rejected |
| 강점 | 기본 태스크 수행 능력 | 미묘한 품질 차이 학습 |
| 위험 | 단순 모방 | 잘못된 선호 강화 |

**지식/포맷은 SFT 또는 RAG, 선호/판단은 DPO.**

### 3.6 복수 chosen 후보를 고려해야 함

법률 검토, 계약 리스크 분석 등은 정답이 하나가 아닐 수 있습니다.

권장 방식:
```
A > B > C > D (ranking 구조)

pair 생성:
A vs C
A vs D
B vs D

A vs B (차이가 너무 미세하면 제외)
```

## 4. 법률/컴플라이언스 모델 Error Taxonomy

| error_type | 설명 | rejected 예시 |
|------------|------|---------------|
| wrong_determination | 판단 자체가 틀림 | 위험 조항을 문제없다고 판단 |
| over_confident | 근거 부족한데 단정 | "명백히 위법입니다" |
| under_refusal | 답변 불가 상황에서 답함 | 문서가 없는데 계약 내용 추정 |
| missing_clause | 핵심 조항 누락 | 해지 조항만 보고 손해배상 조항 누락 |
| irrelevant_clause | 관련 없는 조항 인용 | 개인정보 이슈에 광고 조항 인용 |
| fake_citation | 없는 법령/조항/판례 인용 | 존재하지 않는 제23조 인용 |
| bad_revision | 수정안이 부적절 | 리스크를 줄이지 못하는 대체 문구 |
| format_violation | 요구 형식 위반 | JSON-only 요청에 설명 추가 |
| too_verbose | 불필요하게 장황 | 실무 요약 없이 일반론 과다 |
| too_short | 핵심 근거 누락 | 결론만 있고 이유 없음 |
| terminology_error | 법률 용어 혼동 | 해제/해지/취소 혼용 |
| scope_creep | 질문 범위 초과 | 계약 검토 요청에 세무 자문 확장 |

## 5. 목표 Metric과 DPO 데이터 연결

| 목표 metric | DPO 데이터 방향 |
|-------------|----------------|
| determination_accuracy 개선 | 판단이 갈리는 hard case 중심 pair |
| related_clauses_f1 개선 | 관련 조항 누락/과다 포함 pair |
| related_clauses_precision 개선 | irrelevant clause 인용 rejected |
| related_clauses_recall 개선 | 핵심 조항 누락 rejected |
| valid_citation_metric 개선 | fake citation / unsupported citation rejected |
| format compliance 개선 | JSON/마크다운/mandatory structure 위반 rejected |
| hallucination 감소 | 근거 없는 확정 답변 rejected |
| user utility 개선 | 실무 수정안/요약/리스크 수준 차이 pair |

## 6. 추천 데이터 구성 비율

| 유형 | 비율 | 목적 |
|------|------|------|
| 실제 실패 로그 기반 pair | 40% | 현재 모델 약점 직접 보정 |
| 전문가 작성 hard pair | 25% | 도메인 고급 판단 학습 |
| citation/grounding pair | 15% | 허위 근거 감소 |
| 형식/지시 준수 pair | 10% | 제품 출력 안정화 |
| refusal/uncertainty pair | 10% | 무리한 답변 방지 |

## 7. 데이터 제작 프로세스

1. 현재 모델 eval 실패 케이스 수집
2. 실패 유형 taxonomy 부여
3. 각 prompt에 대해 여러 candidate 생성
4. 전문가 또는 고품질 judge가 ranking
5. chosen/rejected pair 생성
6. 애매한 pair 제거
7. 길이/도메인/난이도/오류유형 분포 점검
8. 소량 DPO 학습
9. held-out eval로 metric별 개선/악화 확인
10. 악화된 metric의 데이터 역추적
11. 다음 라운드 데이터 재구성

**DPO는 한 번에 크게 넣기보다, 작게 넣고 eval → 실패 분석 → 재구성하는 방식이 안전합니다.**

## 8. 망하는 패턴

| 패턴 | 결과 | 해결 |
|------|------|------|
| chosen이 전부 장문 고품질 답변 | 모델이 항상 길게 답함 | 간결하지만 정확한 chosen도 포함 |
| rejected가 너무 말도 안 되는 오답 | 실전 hard case 개선이 적음 | 현재 모델이 실제로 생성한 그럴듯한 오답 사용 |
| 라벨러 기준 불일치 | DPO가 noisy preference를 학습 | rubric, double annotation, disagreement review |
| 법률 근거보다 문체를 우선 | 예쁜 hallucination 증가 | 정확성/근거성을 최상위 기준으로 고정 |
| 평가셋과 유사한 데이터 과다 | eval만 좋아지고 실전 일반화 안 됨 | eval leakage 제거, held-out 도메인 유지 |
| DPO로 지식 주입 시도 | 지식이 안정적으로 안 들어가고 선호만 왜곡 | 지식/포맷은 SFT 또는 RAG, 선호/판단은 DPO |
| base/SFT 모델 수준이 낮은데 DPO부터 함 | chosen을 따라갈 생성 능력이 부족 | SFT 품질을 먼저 확보한 뒤 DPO |

## 9. 고도화 체크리스트

### Prompt 품질
- [ ] 실제 사용 시나리오와 유사한가
- [ ] 충분히 다양한가 (짧은 질문, 긴 문서, 모호한 질문, 복합 지시)
- [ ] 도메인 분포가 목표와 맞는가
- [ ] 난이도 분포가 있는가 (easy/medium/hard/adversarial)
- [ ] 평가셋과 중복되지 않는가

### Chosen 품질
- [ ] 사실 정확성 (조항, 날짜, 수치, 주체)
- [ ] 근거성 (결론이 근거에서 자연스럽게 나오는가)
- [ ] 불확실성 (모호한 경우 단정하지 않는가)
- [ ] 형식 (요구 포맷을 지키는가)
- [ ] 실무성 (사용자가 바로 활용할 수 있는가)
- [ ] 과잉 답변 (묻지 않은 범위까지 확장하지 않는가)

### Rejected 품질
- [ ] 실제 실패와 유사한가 (현재 모델이 만들 법한 오답인가)
- [ ] 너무 쉽지 않은가
- [ ] 오류 유형이 명확한가
- [ ] chosen과 길이 편향이 없는가
- [ ] 위험 행동을 포함하는가 (hallucination, 과잉 확신, 근거 누락)

### Pair 품질
- [ ] 선호 방향이 명확한가 (라벨러 2명이 봐도 같은 결론인가)
- [ ] 차이가 학습 가치 있는가
- [ ] 단일 기준만 다른가
- [ ] 도메인 목표와 연결되는가

### 전체
- [ ] Inter-annotator agreement ≥ 0.7
- [ ] 길이 편향 분석 완료
- [ ] label flip rate < 5% (샘플 검증)
- [ ] SFT policy 분포와 정합성 확인

## 11. 한 줄 결론

> DPO 트레이닝 데이터 고도화의 핵심은 좋은 답변을 많이 넣는 것이 아니라, **모델이 실제로 헷갈리는 그럴듯한 실패 답변**과, **도메인 기준상 더 바람직한 답변의 차이**를 일관되게 학습시키는 것입니다.

## 💡 Online DPO 트렌드

최근 트렌드는 'Online DPO' 방식입니다. 고정된 데이터셋만 쓰지 말고, 현재 훈련 중인 모델이 내놓은 두 답변을 모델(또는 사람)이 실시간으로 평가하여 다시 학습 데이터로 넣는 방식이 성능 고도화에 매우 효과적입니다.
